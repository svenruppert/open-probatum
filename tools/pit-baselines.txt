# Per-package mutation-coverage floors. Format: <package>=<min-percent>

# Floor values are conservative — set 2-3 points below the current
# measurement so casual refactors don't trip the gate, but a meaningful
# regression does. Raise a floor only after demonstrating sustained
# coverage above it; never lower it without a written reason.
#
# Source of truth for "current": target/pit-reports/index.html after
# `./mvnw -o org.pitest:pitest-maven:mutationCoverage`.
#
# Special line `__overall__` enforces the aggregate floor.
#
# Last recalibration: 2026-06-14, after wiring persistent
# JSentinelVersionStore. New packages: .i18n, .security.storage,
# .views.ui. Overall floor lowered from 42 → 35 because the new
# view/UI/i18n code (PushDemoView, SecurityFeaturesView,
# LocaleSwitcher, ThemeSwitcher, BrandMark, …) added ~580 mutants
# in UI assembly code that is intentionally light on assertions —
# the user-facing components are exercised by BrowserlessTest visual
# inspection, not exhaustive mutation kills. Security packages
# still hold above 75 %.
#
# Recalibration 2026-06-25 (V00.10.00 Trust Core, P013): added the new
# domain packages .credential, .catalog, .assessment. Measured (scoped
# PIT): assessment 92 % (12/13), catalog 100 % (1/1, all records),
# credential 67 % (59/88). credential is dragged only by CredentialPdf
# (48 %, PDFBox draw-calls are hard to mutation-kill, like the UI code)
# and CredentialGovernance/ValidationResult thin mutants; the trust-
# critical logic — CredentialValidator, IssuanceService, Credential,
# ValidationRateLimiter — sits at 80–100 %. Floors set 3 below measured.
#
# Recalibration 2026-06-25 (storage-consolidation + minion isolation):
# the jSentinel-00.75.20 migration funnels three Eclipse-Stores into one
# JSentinelStoragePair with a SINGLE lock. Under PITest the forked minions
# (which do not inherit Surefire's app.storage.dir) used to share the
# repo-root ./data and that one lock — a killed, timed-out minion left a
# stale lock the next minion blocked on → full-gate DEADLOCK. Fixed by
# giving each minion its own per-process store (app.storage.isolate, see
# pom _mutation-gate jvmArgs). Consequence: every minion now starts from
# an EMPTY store, so the OLD floors (last full run 2026-06-14, measured in
# the shared-store environment AND before this deadlock fix) were inflated
# by cross-test data accumulation — data-display code paths that only run
# when the store holds data now go uncovered. The floors below are
# recalibrated to the honest isolated measurement (full gate, 1088
# mutants). The lower security/security.storage numbers are structural:
# the new IODH AppStorage singleton's open() path and the servlet
# SecurityHeadersFilter are hard to mutation-test (the same reason the
# other providers sit where they do); the trust + auth LOGIC is verified
# by 288 green tests + exit-review #2, not by these floors alone.
